## 2. 查询API (`/query`) 功能审计

### 2.1 核心功能
✅ **已实现功能**:
- 同步查询 (`POST /query`)
- 流式查询 (`POST /query/stream`, `POST /stream`)
- 会话上下文管理
- 多Agent路由 (auto/cybersecurity/ai/pdf_text/general)
- 检索策略切换 (baseline/advanced/safe)
- Web回退机制
- 推理模式 (reasoning)
- PDF文档专用处理
- 文件清单查询策略

### 2.2 发现的问题

#### 🔴 严重问题

**P0-1: 流式查询中断后状态不一致**
- **位置**: `app/api/routes/query.py:722-856`
- **问题**: 当用户中断流式查询时，后端可能已写入部分消息到会话历史，但前端显示不完整
- **影响**: 会话历史污染，用户体验差
- **建议**: 
  - 实现事务性消息写入
  - 中断时标记消息为 `interrupted` 状态
  - 前端显示中断标记

**P0-2: 缓存键冲突风险**
- **位置**: `app/api/routes/query.py:262-271`
- **问题**: 缓存键生成未包含 `memory_context`，可能导致不同上下文返回相同缓存结果
- **影响**: 答案不准确，用户困惑
- **建议**: 将 `memory_context` 哈希值加入缓存键

**P0-3: PDF Agent 逻辑分散**
- **位置**: `query.py:146-217` 和 `query.py:515-625`
- **问题**: PDF处理逻辑在同步和流式端点重复，维护困难
- **影响**: 代码重复，bug修复需要两处同步
- **建议**: 提取为独立函数 `_handle_pdf_agent_routing()`

#### 🟡 中等问题

**P1-1: 错误处理不统一**
- **位置**: 多处 try-except 块
- **问题**: 
  - 有些返回 HTTPException
  - 有些返回错误消息字符串
  - 流式查询错误通过SSE事件返回
- **建议**: 统一错误响应格式，定义标准错误码

**P1-2: 超时配置硬编码**
- **位置**: `query.py:754`
- **问题**: `timeout_ms=20000` 硬编码在代码中
- **建议**: 移到配置文件，支持按用户角色/查询类型动态调整

**P1-3: 重复的一致性检查逻辑**
- **位置**: `query.py:316-346`
- **问题**: 一致性守卫逻辑只在同步查询中实现，流式查询缺失
- **影响**: 流式查询可能返回不一致答案
- **建议**: 提取为独立服务，两种查询模式共用

#### 🟢 轻微问题

**P2-1: 日志记录不足**
- **问题**: 关键决策点缺少日志 (如Agent选择、策略降级)
- **建议**: 增加结构化日志，便于问题排查

**P2-2: 性能指标不完整**
- **问题**: 只记录总延迟，缺少各阶段细分 (检索/重排/生成)
- **建议**: 增加分段计时，输出到 `debug.timing` 字段

## 3. 会话管理API (`/sessions`) 审计

### 3.1 核心功能
✅ **已实现**:
- 会话CRUD (创建/读取/删除)
- 消息编辑和删除
- 长期记忆管理
- 策略锁定 (strategy_lock)

### 3.2 发现的问题

#### 🟡 中等问题

**P1-4: 会话并发安全**
- **位置**: `sessions.py:127-184`
- **问题**: 消息更新和重新运行没有并发锁，可能导致竞态条件
- **影响**: 多个请求同时修改同一会话时数据损坏
- **建议**: 使用分布式锁或乐观锁机制

**P1-5: 策略锁定未在查询中强制执行**
- **位置**: `sessions.py:70-94`
- **问题**: 设置了 `strategy_lock` 但查询端点未检查
- **影响**: 功能无效
- **建议**: 在 `_effective_strategy_for_session()` 中优先使用锁定策略

#### 🟢 轻微问题

**P2-3: 会话列表无分页**
- **问题**: `GET /sessions` 返回所有会话，用户会话多时性能差
- **建议**: 增加分页参数 (limit/offset)

## 4. 文档管理API (`/documents`) 审计

### 4.1 核心功能
✅ **已实现**:
- 文档上传 (多文件、进度跟踪)
- 文档删除 (索引/物理文件)
- 文档重新索引
- 可见性控制 (private/public)
- Agent分类自动推断

### 4.2 发现的问题

#### 🔴 严重问题

**P0-4: 文件上传无病毒扫描**
- **位置**: `documents.py:132-252`
- **问题**: 直接保存用户上传文件，无安全检查
- **影响**: 恶意文件可能感染服务器
- **建议**: 
  - 集成 ClamAV 或云端扫描服务
  - 隔离上传文件到临时目录
  - 扫描通过后再移动到正式目录

**P0-5: 文件签名验证不完整**
- **位置**: `documents.py:193-196`
- **问题**: 只检查前16字节，容易绕过
- **影响**: 恶意文件伪装成PDF上传
- **建议**: 
  - 使用 `python-magic` 库深度检测
  - 验证完整文件结构
  - 增加文件大小合理性检查

#### 🟡 中等问题

**P1-6: 上传进度不准确**
- **位置**: `documents.py:168-181`
- **问题**: 只统计上传字节数，不包含索引时间
- **影响**: 进度条到100%后还要等待很久
- **建议**: 分两阶段显示进度 (上传50% + 索引50%)

**P1-7: 删除操作无审计日志**
- **位置**: `documents.py:54-75`
- **问题**: 删除文件时只记录简单审计，缺少详细信息
- **建议**: 记录文件哈希、大小、删除原因

**P1-8: 重新索引无版本控制**
- **位置**: `documents.py:78-130`
- **问题**: 重新索引直接覆盖旧数据，无法回滚
- **建议**: 保留旧版本索引，支持版本切换

#### 🟢 轻微问题

**P2-4: 上传文件名未清理**
- **问题**: 直接使用用户提供的文件名，可能包含特殊字符
- **建议**: 清理文件名，移除路径遍历字符

**P2-5: 缺少批量操作API**
- **问题**: 删除多个文档需要多次请求
- **建议**: 增加批量删除端点

## 5. 管理员API (`/admin/*`) 审计

### 5.1 运维管理 (`/admin/ops`)

✅ **已实现功能**:
- 系统概览 (KPI、错误率、慢请求)
- 审计日志导出 (CSV/Markdown)
- 检索策略切换
- 灰度发布 (Canary)
- 基准测试
- A/B对比测试
- 自动调优
- 一键回滚

#### 🟡 中等问题

**P1-9: 自动调优逻辑过于简单**
- **位置**: `admin_ops.py:546-584`
- **问题**: 只根据单次趋势调整参数，可能过度反应
- **建议**: 
  - 使用滑动窗口平均值
  - 设置调整幅度上限
  - 增加回退机制

**P1-10: 基准测试阻塞请求**
- **位置**: `admin_ops.py:383-441`
- **问题**: 基准测试同步执行，可能耗时很久
- **建议**: 改为后台任务，返回任务ID供轮询

**P1-11: 灰度发布无健康检查**
- **位置**: `admin_ops.py:334-353`
- **问题**: 切换策略后不自动检查指标是否恶化
- **建议**: 增加自动回滚机制，监控错误率和延迟

#### 🟢 轻微问题

**P2-6: 慢请求列表固定10条**
- **位置**: `admin_ops.py:112`
- **问题**: 硬编码 `[:10]`，无法调整
- **建议**: 增加 `limit` 参数

### 5.2 用户管理 (`/admin/users`)

✅ **已实现功能**:
- 用户列表和筛选
- 角色/状态更新
- 用户分类管理
- 密码重置 (需审批)
- 管理员创建 (需审批令牌)

#### 🔴 严重问题

**P0-6: 管理员审批令牌存储不安全**
- **位置**: 数据库表设计
- **问题**: `admin_approval_token_hash` 使用简单哈希，易被彩虹表攻击
- **建议**: 
  - 使用 bcrypt/argon2 加盐哈希
  - 增加令牌过期时间
  - 限制尝试次数

**P0-7: 密码重置无二次验证**
- **位置**: 管理员可直接重置任何用户密码
- **问题**: 单点失陷风险高
- **建议**: 
  - 增加邮件/短信验证
  - 要求两名管理员审批
  - 记录详细审计日志

#### 🟡 中等问题

**P1-12: 用户在线状态判断不准确**
- **问题**: 基于 `last_seen_at` 判断，但未考虑时区
- **建议**: 统一使用UTC时间，增加心跳机制

## 6. 认证API (`/auth`) 审计

### 6.1 核心功能
✅ **已实现**:
- 用户注册
- 登录/登出
- 会话管理
- 速率限制
- Cookie认证

### 6.2 发现的问题

#### 🔴 严重问题

**P0-8: 登录速率限制可绕过**
- **位置**: `auth.py:46`
- **问题**: 基于 `IP + username` 限制，攻击者可换用户名绕过
- **建议**: 
  - 增加全局IP限制
  - 使用滑动窗口算法
  - 增加验证码

**P0-9: Token无刷新机制**
- **问题**: Token过期后需要重新登录，用户体验差
- **建议**: 实现 refresh token 机制

#### 🟡 中等问题

**P1-13: 注册无邮箱验证**
- **问题**: 任何人都可以注册，易被滥用
- **建议**: 
  - 增加邮箱验证
  - 或改为邀请制

**P1-14: 密码强度检查在前端**
- **位置**: 后端未验证密码复杂度
- **问题**: 前端检查可绕过
- **建议**: 后端增加密码策略验证

## 8. API通用问题

### 8.1 跨领域问题

#### 🔴 严重问题

**P0-10: CORS配置过于宽松**
- **问题**: 可能允许任意源访问
- **建议**: 严格限制允许的源

**P0-11: 无API版本控制**
- **问题**: 破坏性变更会影响现有客户端
- **建议**: 实现 `/v1/`, `/v2/` 版本前缀

#### 🟡 中等问题

**P1-15: 无全局请求ID**
- **问题**: 难以追踪跨服务请求
- **建议**: 增加 `X-Request-ID` 头

**P1-16: 响应时间未标准化**
- **问题**: 有些返回毫秒，有些返回秒
- **建议**: 统一使用毫秒

**P1-17: 分页参数不一致**
- **问题**: 有些用 `limit/offset`，有些用 `page/size`
- **建议**: 统一分页参数命名

#### 🟢 轻微问题

**P2-9: 缺少API文档**
- **问题**: 虽然有OpenAPI，但缺少详细说明和示例
- **建议**: 完善Swagger文档

**P2-10: 响应格式不统一**
- **问题**: 成功响应格式各异
- **建议**: 定义标准响应包装器 `{success, data, error, meta}`

## 9. 性能问题

### 9.1 数据库查询

**P1-18: N+1查询问题**
