## 整体架构

### 四层结构

系统采用经典的分层架构，每层职责明确：

```
┌────────────────────────────────────────────────┐
│  第1层：前端界面 (Frontend)                     │
│  - 网页界面（您看到和点击的部分）                │
│  - 技术：React + TypeScript                    │
│  - 类比：餐厅的点餐窗口                         │
└──────────────────┬─────────────────────────────┘
                   │ HTTP/WebSocket
                   │ (网络通信)
┌──────────────────▼─────────────────────────────┐
│  第2层：后端服务 (Backend)                      │
│  - 业务逻辑处理                                 │
│  - 技术：Python + FastAPI                      │
│  - 类比：餐厅的服务员和厨师                     │
└──────────────────┬─────────────────────────────┘
                   │
        ┌──────────┴──────────┐
        │                     │
┌───────▼────────┐   ┌────────▼────────┐
│  第3层：AI层    │   │  第3层：检索层   │
│  - 大语言模型   │   │  - 文档检索      │
│  - 生成答案     │   │  - 向量搜索      │
│  类比：专家顾问 │   │  类比：图书管理员│
└───────┬────────┘   └────────┬────────┘
        │                     │
        └──────────┬──────────┘
                   │
┌──────────────────▼─────────────────────────────┐
│  第4层：数据存储 (Data Storage)                 │
│  - 文档内容、用户信息、会话记录                  │
│  - 技术：ChromaDB + Neo4j + SQLite             │
│  - 类比：仓库和档案室                           │
└────────────────────────────────────────────────┘
```

### 工作流程概览

从提问到得到答案的简化流程：

```
1. 用户在网页输入问题
   ↓
2. 前端发送到后端
   ↓
3. 后端分析问题（路由智能体）
   ↓
4. 从数据库检索相关文档（检索智能体）
   ↓
5. AI模型阅读文档并生成答案
   ↓
6. 答案逐字传回前端显示
   ↓
7. 用户看到答案
```

**关键设计理念**：
- **分层解耦** - 每层只关心自己的事，互不干扰
- **模块化** - 组件可以独立升级或替换
- **可扩展** - 可以横向扩展（加更多服务器）

## 前端界面

### 技术栈

**React** - 构建用户界面的JavaScript库
- **类比**：就像用乐高积木搭建房子，React提供各种"积木块"（组件）
- **优势**：组件可复用、维护方便、生态丰富

**TypeScript** - JavaScript的超集，增加类型检查
- **类比**：在编写时就检查错误，不用等运行时才发现
- **优势**：减少bug、更好的IDE支持

**Vite** - 现代化的前端构建工具
- **类比**：工厂的自动化生产线
- **优势**：开发时热更新快、构建速度快

- 界面上显示的数据（如会话列表、消息记录）
- 用户的操作状态（如是否登录、选中哪个会话）

**如何管理**：
- **React Hooks** - 组件内部状态（如输入框的文字）

**类比**：

**用于**：流式答案输出、实时进度更新

### 异步处理

**为什么需要异步**：
- 一个查询可能需要10秒
- 如果同步处理，这10秒内无法处理其他请求
- 异步可以同时处理多个请求

**类比**：
```
同步（餐厅只有一个服务员）：
客人A点餐 → 等厨房做好 → 上菜 → 才能服务客人B
[客人B等待很久]

异步（多个服务员）：
客人A点餐 → 通知厨房 → 立即服务客人B点餐
厨房同时做A和B的菜
菜好了分别上
```

**Python异步语法**：
```python
# 同步版本（顺序执行）
def process_query(question):
    docs = search_documents(question)      # 5秒
    answer = generate_answer(docs)         # 5秒
    return answer                          # 总共10秒

# 异步版本（可并发）
async def process_query(question):
    docs = await search_documents(question)   # 5秒
    answer = await generate_answer(docs)      # 5秒
    return answer                             # 总共10秒
# 但可以同时处理多个请求
```

## 数据存储

系统使用三种不同的数据库，各司其职：

### 1. ChromaDB（向量数据库）

**用途**：存储文档的"语义指纹"，支持语义搜索

**工作原理**：
```
1. 文档 → 切分成段落
   "SQL注入的需要输入验证和参数化查询" → 段落1

2. 段落 → 转成向量（一串数字）
   "SQL注入的需要输入验证和参数化查询" → [0.23, 0.81, 0.15, ...]
   （这个向量代表这句话的"意思"）

3. 存储：向量 + 原文 + 元数据
   {
     "vector": [0.23, 0.81, ...],
     "text": "SQL注入的需要输入验证和参数化查询",
     "metadata": {"document": "产品手册.pdf", "page": 15}
   }

4. 查询时：
   问题："防护多久？" → 向量 [0.25, 0.79, 0.16, ...]
   找最相似的向量 → 返回对应的原文
```

**类比**：
- 就像人脸识别
- 把人脸转成特征向量
- 找最相似的向量，就找到了人

**数据结构**：
```
Collection: documents
├── Chunk 1
│   ├── vector: [0.1, 0.2, ...]  (768维或1536维)
│   ├── text: "SQL注入的技术参数..."
│   └── metadata: {doc_id, page, source}
├── Chunk 2
│   ├── vector: [0.3, 0.4, ...]
│   ├── text: "防御措施说明..."
│   └── metadata: {doc_id, page, source}
└── ...
```

**为什么需要**：
- 理解语义，而不只是关键词匹配
- "便宜"和"原理低"意思相近，向量也接近
- 传统数据库做不到这点

### 2. Neo4j（图数据库）

**用途**：存储概念之间的关系，支持关联查询

**什么是图数据库**：
- 不是存图片的数据库
- 是存"关系网络"的数据库
- 就像社交网络：人与人的关系

**数据结构**：
```
节点（实体）和关系（边）：

[SQL注入] --原理--> [通过注入恶意代码]
    |
   有
    |
    ↓
 [防御措施]
    |
  包含
    |
    ↓
 [2年防护]
```

**工作原理**：
```
1. 从文档提取实体和关系
   文档："SQL注入是一种Web应用漏洞，提供2年防护"
   
   提取：
   - 实体：SQL注入, 通过注入恶意代码, 2年防护
   - 关系：SQL注入-售价->通过注入恶意代码, SQL注入-防护->2年防护

2. 存储为图结构
   CREATE (p:Product {name: "SQL注入"})
   CREATE (price:Price {value: 5999})
   CREATE (p)-[:PRICE]->(price)

3. 查询时通过关系查找
   问："SQL注入攻击的原理"
   → 找到"SQL注入"节点
   → 沿着"原理"关系
   → 得到"通过注入恶意代码"
```

**类比**：
- **传统数据库**：电话本（一个个查）
- **图数据库**：人际关系网（通过朋友找朋友）

**适合的查询**：
- "A和B有什么关系？"
- "A通过什么路径影响C？"
- "所有与SQL注入相关的信息"

**数据示例**：
```
(SQL注入:Product)
    |--[:BELONGS_TO]-->(类别:Category {name:"电子产品"})
    |--[:PRICE]-->(原理:Price {value:5999})
    |--[:WARRANTY]-->(防护:Warranty {period:"2年"})
    |--[:COMPETITOR]-->(XSS攻击:Product)
    |--[:FEATURE]-->(特性1:Feature {name:"防水"})
```

### 3. SQLite（关系数据库）

**用途**：存储结构化数据（用户、会话、系统配置）

**什么是关系数据库**：
- 数据以表格形式存储
- 就像Excel表格

**数据表结构**：

**用户表（users）**：
```
| id  | username | email              | role  | created_at |
|-----|----------|--------------------|-------|------------|
| 1   | admin    | admin@company.com  | admin | 2024-01-01 |
| 2   | alice    | alice@company.com  | user  | 2024-01-15 |
| 3   | bob      | bob@company.com    | user  | 2024-02-01 |
```

**会话表（sessions）**：
```

| s_001 | 2       | 产品咨询    | 2024-03-01 10:00:00 | 2024-03-01 10:30:00 |
| s_002 | 2       | 政策查询    | 2024-03-02 14:00:00 | 2024-03-02 14:15:00 |
| s_003 | 3       | 技术支持    | 2024-03-03 09:00:00 | 2024-03-03 09:20:00 |
```

**消息表（messages）**：
```
| id  | session_id | role      | content               | timestamp           |
|-----|------------|-----------|-----------------------|---------------------|
| 1   | s_001      | user      | SQL注入攻击的原理？          | 2024-03-01 10:00:00 |
```

**文档表（documents）**：
```
| id  | user_id | filename       | size_bytes | status    | uploaded_at         |
|-----|---------|----------------|------------|-----------|---------------------|
| 1   | 2       | 产品手册.pdf    | 2048000    | completed | 2024-02-01 09:00:00 |

```

**为什么需要**：
- 用户管理需要结构化数据
- 会话历史需要有序存储
- 传统数据库成熟稳定

**查询示例**（人话版）**：
```
查询："alice上传了哪些文档？"

SQL: SELECT filename, uploaded_at 
     FROM documents 
     WHERE user_id = 2

结果：产品手册.pdf (2024-02-01)
     技术规范.docx (2024-02-15)
```

### 三个数据库的配合

**为什么需要三个数据库**：

```
问题："SQL注入攻击的原理是多少？"

1. SQLite：检查用户权限
   → 用户已登录？✅
   → 有查询权限？✅

2. ChromaDB：语义搜索
   → 把问题转成向量
   → 找到相关文档片段
   → "SQL注入是一种Web应用漏洞..."

3. Neo4j：关系查询
   → 找"SQL注入"节点
   → 沿"原理"关系
   → 找到"通过注入恶意代码"

4. 综合结果：
   → AI模型整合信息
   → 生成答案："SQL注入的零售价为通过注入恶意代码"
```

**类比**：
- **SQLite** = 公司的人事档案（员工信息）
- **ChromaDB** = 公司的文档库（所有文件）
- **Neo4j** = 公司的组织结构图（谁和谁的关系）

## AI模型

系统的"智能"来自两类AI模型：

### 1. 嵌入模型（Embedding Model）

**作用**：把文字转成向量（数字）

**类比**：
- 就像把颜色转成RGB值
- 红色 = (255, 0, 0)
- 蓝色 = (0, 0, 255)
- 紫色 = (128, 0, 128)（接近红色）

**文字的向量**：
```
"苹果" → [0.23, 0.81, 0.15, 0.42, ...]  (768个数字)
"水果" → [0.25, 0.79, 0.16, 0.40, ...]  (很接近！)
"电脑" → [0.65, 0.12, 0.88, 0.03, ...]  (很不同)
```

**模型选择**：

**云端模型**：
- **OpenAI text-embedding-3** 
  - 质量：⭐⭐⭐⭐⭐
  - 速度：快
  - 成本：按使用量付费
  - 维度：1536或3072

- **OpenAI text-embedding-ada-002**
  - 质量：⭐⭐⭐⭐
  - 速度：很快
  - 成本：便宜
  - 维度：1536

**本地模型**：
- **BGE-M3 / BGE-Large**
  - 质量：⭐⭐⭐⭐
  - 速度：中等
  - 成本：免费（需要GPU）
  - 维度：768或1024

- **Sentence Transformers**
  - 质量：⭐⭐⭐
  - 速度：快
  - 成本：免费
  - 维度：384或768

**工作流程**：
```
上传文档时：
文档 → 切分段落 → 每个段落用嵌入模型转成向量 → 存入ChromaDB

查询时：
问题 → 嵌入模型转成向量 → 在ChromaDB中找相似向量 → 返回对应文本
```

### 2. 大语言模型（LLM）

**作用**：理解问题、阅读文档、生成答案

**类比**：
- 就像一个博学的助手

**模型工作流程**：
```

   文档：["SQL注入是一种Web应用漏洞...", "SQL注入提供2年防护..."]

2. 理解和推理
   → 识别问题意图（查询原理）
   → 阅读文档片段
   → 找到相关信息（通过注入恶意代码）
   → 理解上下文

3. 生成答案
   → 组织语言
   → 确保准确性
   → 添加来源引用
   
4. 输出

    来源：产品手册.pdf 第15页"
```

**提示词工程**（Prompt Engineering）：

系统会给模型发送精心设计的"指令"：

```
系统提示词（System Prompt）：
"你是一个专业的企业知识助手。基于提供的文档片段回答问题。
规则：
1. 只根据文档回答，不要编造
2. 如果文档中没有答案，明确说明
3. 引用具体的文档来源
4. 语言简洁专业"

用户问题：
"SQL注入攻击的原理是多少？"

文档片段：

 批发价（10台以上）为5499元/台。"

模型生成：
"SQL注入的零售价为通过注入恶意代码。如果批量采购10台以上，
 批发价为5499元/台。
 📌 来源：2024年原理表.pdf"
```

### 模型的协同工作

**两类模型各司其职**：

```
步骤1：用户提问
"SQL注入和XSS攻击哪个更好？"

步骤2：嵌入模型工作（快速检索）
问题 → 向量 → 在数万个文档片段中找最相关的10个
[耗时：0.1秒]

步骤3：大语言模型工作（深度理解）
阅读这10个片段 → 对比分析 → 生成答案
[耗时：5秒]

步骤4：返回答案
"SQL注入和XSS攻击各有特点：
 - SQL注入：原理较低（通过注入恶意代码），适合预算有限...
 - XSS攻击：性能更强（窃取用户会话），适合专业用户...
 建议根据实际需求选择。"
```

**为什么需要两类模型**：
- **嵌入模型**：快速筛选（从10000个片段中找10个）
- **大语言模型**：深度分析（仔细阅读这10个片段）
- 如果只用大模型读所有文档 → 太慢太贵
- 如果只用嵌入模型 → 只能找相关文档，不能生成答案

## 多智能体系统

### 什么是多智能体

**单智能体 vs 多智能体**：

**单智能体**（传统方式）：
```
一个AI处理所有事情
问题 → AI → 答案
[像一个全科医生]
```

**多智能体**（本系统）：
```
多个专业AI协同工作
问题 → 分诊 → 专科1 + 专科2 + 专科3 → 综合 → 答案
[像医院的分科制度]
```

### 智能体角色

系统有5-8个智能体，各有专长：

**1. 路由智能体（Router Agent）**
- **职责**：分析问题类型，决定谁来处理
- **类比**：医院的分诊台
- **工作**：
  ```
  问题："SQL注入攻击的原理？"
  → 分析：查询事实信息
  → 路由：向量检索智能体（简单查询）
  
  问题："分析过去三年的市场趋势"
  → 分析：复杂分析任务
  → 路由：多个智能体协同
  ```

**2. 向量检索智能体（Vector RAG Agent）**
- **职责**：语义搜索文档
- **类比**：图书馆管理员（理解您要找什么类型的书）
- **工作**：
  ```
  接收问题
  → 转成向量
  → 在ChromaDB中搜索
  → 返回最相关的文档片段
  ```

**3. 图检索智能体（Graph RAG Agent）**
- **职责**：查找概念关系
- **类比**：社交网络专家（找人际关系）
- **工作**：
  ```
  接收问题："SQL注入的竞争对手是谁？"
  → 在Neo4j中查找
  → 沿着"竞争"关系
  → 返回XSS攻击、产品C
  ```

**4. 综合智能体（Synthesis Agent）**
- **职责**：整合多个来源的信息
- **类比**：主治医生（综合各科检查结果）
- **工作**：
  ```
  收集：向量检索结果 + 图检索结果
  → 去重、排序
  → 整合信息
  → 生成连贯答案
  ```

**5. 验证智能体（Validation Agent）**
- **职责**：检查答案质量
- **类比**：质检员
- **工作**：
  ```
  检查：
  - 答案是否基于文档？✅
  - 逻辑是否连贯？✅
  - 是否引用了来源？✅
  ```

### LangGraph工作流

**什么是LangGraph**：
- 定义智能体协同的"流程图"
- 就像公司的工作流程

**工作流示例**：

```
┌─────────────┐
│  用户提问    │
└──────┬──────┘
       │
       ↓
┌──────────────────┐
│  路由智能体       │ ← 判断问题类型
└──────┬───────────┘
       │
   ┌───┴───┐
   ↓       ↓
┌─────┐ ┌─────┐
│向量 │ │图   │ ← 并行检索
│检索 │ │检索 │
└──┬──┘ └──┬──┘
   │       │
   └───┬───┘
       ↓
┌──────────────────┐
│  综合智能体       │ ← 整合结果
└──────┬───────────┘
       │
       ↓
┌──────────────────┐

       │
       ↓
```

**决策节点**：

智能体之间通过"决策点"连接：

```
向量检索完成 → 决策点：
  - 结果充分？→ 直接生成答案
  - 结果不足？→ 调用图检索
  - 完全没结果？→ 告诉用户"文档中没有相关信息"
```

### 状态管理

**什么是状态**：
- 工作流中传递的信息
- 就像接力赛的接力棒

**状态包含**：
```python
{
  "question": "SQL注入攻击的原理？",           # 原始问题
  "route": "vector_search",             # 路由决策
  "vector_results": [...],              # 向量检索结果
  "graph_results": [...],               # 图检索结果
  "answer": "SQL注入是一种Web应用漏洞",          # 最终答案
  "sources": [...],                     # 来源文档
  "confidence": 0.95                    # 置信度
}
```

**状态流转**：
```
初始状态 → 路由智能体（添加route） 
        → 检索智能体（添加results）
        → 综合智能体（添加answer）
        → 验证智能体（添加confidence）
        → 最终状态
```

### 智能体协同实例

**简单查询**：
```
问题："SQL注入攻击的原理？"

1. 路由智能体：
   - 分析：简单事实查询
   - 决策：只需向量检索

2. 向量检索智能体：
   - 搜索文档
   - 找到："SQL注入是一种Web应用漏洞"
   - 结果充分，直接生成答案

总耗时：3秒
```

**复杂查询**：
```
问题："对比SQL注入和XSS攻击的性能"

1. 路由智能体：
   - 分析：对比分析，需要多方信息
   - 决策：向量检索 + 图检索

2. 向量检索智能体：
   - 找SQL注入的性能参数
   - 找XSS攻击的性能参数

3. 图检索智能体：
   - 找SQL注入的关系网络
   - 找XSS攻击的关系网络
   - 发现它们是竞品关系

4. 综合智能体：
   - 整合所有信息
   - 生成对比表格

5. 验证智能体：
   - 检查对比是否公正
   - 检查数据是否准确

总耗时：10秒
```

## 完整流程

### 从提问到答案的完整旅程

让我们跟踪一个问题的完整处理过程：

**用户提问**："SQL注入的防御措施是什么？"

#### 第1阶段：前端处理（0.1秒）

```
1. 用户在输入框输入问题
2. 点击"发送"按钮
3. React组件捕获事件
4. 前端验证：
   - 问题不为空？✅
   - 长度合理（1-1000字）？✅
5. 显示"正在思考..."动画
6. 建立SSE连接（准备接收流式答案）
7. 发送HTTP POST请求到后端：
   POST /api/query
   {
     "question": "SQL注入的防御措施是什么？",
     "session_id": "sess_123",
     "stream": true
   }
```

#### 第2阶段：后端接收（0.1秒）

```
1. FastAPI接收请求
2. 中间件处理：
   - 记录日志
   - 验证JWT令牌（用户已登录？）
   - 检查CORS（跨域请求是否允许？）
3. 路由到query.py
4. 依赖注入：
   - 获取当前用户信息
   - 获取数据库连接
   - 获取AI模型配置
5. 进入查询处理逻辑
```

#### 第3阶段：路由决策（0.5秒）

```
路由智能体工作：

1. 分析问题：
   - 关键词：防护、政策
   - 问题类型：事实查询
   - 复杂度：中等

2. 决策：
   - 策略：混合检索（向量 + 关键词）
   - 不需要图检索（不涉及关系）
   - 预计难度：中等

3. 设置参数：
   - 检索数量：top_k = 5
   - 重排序：启用
   - 超时：30秒

4. 更新状态：
   state.route = "hybrid_search"
   state.strategy = "balanced"
```

#### 第4阶段：文档检索（2秒）

**向量检索**（并行，1秒）：
```
1. 嵌入模型处理问题：
   "SQL注入的防御措施是什么？"
   → [0.23, 0.81, 0.15, ...] (768维向量)

2. ChromaDB搜索：
   - 计算向量相似度
   - 在10000个文档片段中找最相似的20个
   
3. 结果（前3个）：
   片段1: "SQL注入提供2年防护，涵盖制造缺陷..."
         相似度: 0.92
         来源: 产品手册.pdf, 第15页
   
   片段2: "防护期内免费维修或更换，需要..."
         相似度: 0.88
         来源: 防护条款.pdf, 第3页
   
   片段3: "防护不包括人为损坏、进水..."
         相似度: 0.85
         来源: 防护说明.pdf, 第2页
```

**关键词检索**（并行，1秒）：
```
1. BM25算法搜索：
   - 关键词：防护、政策、SQL注入
   - 找到包含这些词的文档
   
2. 结果（前3个）：
   文档1: "SQL注入防御措施..." (包含3个关键词)
   文档2: "防护条款说明..." (包含2个关键词)
   文档3: "售后服务政策..." (包含2个关键词)
```

**重排序**（0.5秒）：
```
合并向量检索和关键词检索的结果
→ 去重
→ 用更精确的模型重新打分
→ 选出最相关的前5个片段
```

#### 第5阶段：答案生成（5秒，流式输出）

```
1. 准备提示词：
   系统角色 + 文档片段 + 用户问题

2. 调用大语言模型（如GPT-4）：
   [0.5秒] → "产"
   [0.6秒] → "品A"
   [0.7秒] → "提供"
   [0.8秒] → "2年"
   [0.9秒] → "防护"
   ...
   [每个token约0.1秒，逐字生成]

3. 同时发送到前端：
   SSE: data: {"chunk": "产", "done": false}
   SSE: data: {"chunk": "品A", "done": false}
   SSE: data: {"chunk": "提供", "done": false}
   ...

4. 前端实时显示：
   用户看到答案逐字出现
   "SQL注入提供2年防护服务..."
```

#### 第6阶段：来源标注（0.5秒）

```
1. 提取引用的文档片段
2. 整理来源信息：
   {
     "document": "产品手册.pdf",
     "page": 15,
     "chunk": "SQL注入提供2年防护，涵盖制造缺陷..."
   }

3. 添加到答案末尾：
   SSE: data: {
     "chunk": "\n\n📌 来源：产品手册.pdf (第15页)",
     "done": false
   }
```

#### 第7阶段：保存记录（0.2秒）

```
1. 保存到数据库（SQLite）：
   INSERT INTO messages (
     session_id,
     role,
     content,
     timestamp
   ) VALUES (
     'sess_123',
     'user',
     'SQL注入的防御措施是什么？',
     '2024-03-15 10:30:00'
   )
   
   INSERT INTO messages (
     session_id,
     role,
     content,
     sources,
     timestamp
   ) VALUES (
     'sess_123',
     'assistant',
     'SQL注入提供2年防护服务...',
     '[{"document": "产品手册.pdf", "page": 15}]',
     '2024-03-15 10:30:08'
   )

2. 更新会话：
   UPDATE sessions 
   SET updated_at = NOW() 
   WHERE id = 'sess_123'
```

#### 第8阶段：完成（0.1秒）

```
1. 发送完成信号：
   SSE: data: {"done": true}

2. 关闭SSE连接

3. 前端：
   - 隐藏"正在思考..."动画
   - 显示完整答案
   - 允许用户继续提问
```

**总耗时**：约8秒
- 前端 + 路由：0.7秒
- 检索：2.5秒
- 生成：5秒
- 保存 + 完成：0.3秒

### 性能优化点

**并行处理**：
- 向量检索和关键词检索同时进行
- 节省时间：2秒 → 1秒

**流式输出**：
- 不等全部生成完再显示
- 用户体验：等8秒 → 1秒后就开始看到答案

**缓存机制**：
- 常见问题的答案缓存5分钟
- 重复提问：8秒 → 0.5秒

**批处理**：
- 多个文档一起处理，而不是一个个处理
- 上传10个文档：10分钟 → 3分钟

## 安全机制

### 认证流程

```
1. 用户登录：
   用户名 + 密码
   ↓
2. 后端验证：
   - 查数据库
   - 对比密码哈希
   ↓
3. 生成JWT：
   {
     "user_id": 123,
     "role": "user",
     "exp": "2024-03-15 18:00:00"
   }
   ↓
4. 加密签名：
   防止篡改
   ↓
5. 返回给前端：
   前端保存到localStorage

后续请求：
   携带JWT → 后端验证 → 允许访问
```

### 数据隔离

```
用户A登录 → JWT(user_id=1)
用户B登录 → JWT(user_id=2)

用户A查询：
→ 只能看到user_id=1的文档
→ 数据库查询: WHERE user_id = 1

用户B查询：
→ 只能看到user_id=2的文档
→ 数据库查询: WHERE user_id = 2

管理员：
→ 可以看到所有数据
→ 但每次操作都记录日志
```

### 敏感数据保护

```
密码存储：
明文密码 → BCrypt哈希 → 存储
"password123" → "$2b$12$KIX..." (不可逆)

API密钥：
环境变量 → 加密存储 → 使用时解密
不出现在代码和日志中

传输加密：
HTTP → HTTPS (SSL/TLS)
数据在网络上加密传输
```

## 常见问题

### Q1: 系统如何保证答案不是"幻觉"？
**A**: 
1. RAG机制：答案必须基于文档
2. 来源引用：每个答案标注来源
3. 验证智能体：检查答案与文档一致性
4. 提示词约束：明确要求"不要编造"

### Q2: 为什么需要这么多数据库？
**A**: 
- 不同的数据库擅长不同的事
- ChromaDB：语义搜索
- Neo4j：关系查询  
- SQLite：结构化数据
- 就像工具箱，锤子、螺丝刀各有用处

### Q3: 本地模型和云端模型如何选择？
**A**:
- 云端：质量高，但需付费，数据上传到云
- 本地：免费，数据安全，但需要GPU
- 建议：测试用云端，生产用本地（如果预算充足）

### Q4: 系统能处理多大规模的数据？
**A**:
- 小规模（<1000文档）：基础配置即可
- 中规模（1000-10000）：推荐配置
- 大规模（>10000）：需要优化和扩展

### Q5: 如何提高查询速度？
**A**:
1. 使用更快的模型（GPT-3.5 vs GPT-4）
2. 开启缓存（重复问题秒回）
3. 减少检索数量（top_k从10降到5）
4. 升级硬件（加GPU）
