## 📋 变更概述

### 问题描述

v0.2.5之后，项目代码存在严重的**单体文件问题**：

1. **主要问题文件**（共9,135行代码）：
   - `app/api/main.py`: 4,150行（路由、依赖、中间件混在一起）
   - `app/services/auth_service.py`: 930行（认证逻辑庞大）
   - `app/services/workflow_manager.py`: 532行（工作流逻辑复杂）
   - `app/services/hybrid_retriever.py`: 512行（检索逻辑耦合）
   - `app/services/loaders.py`: 508行（多种加载器混合）
   - `app/graph/streaming.py`: 503行（流式处理和SSE混合）
   - `app/api/dependencies.py`: 大量依赖混在一起

2. **代码质量问题**：
   - 单个文件职责不清
   - 代码难以测试和维护
   - 修改风险高，影响范围大
   - 新功能添加困难
   - 代码复用率低

3. **技术债务**：
   - 没有清晰的模块边界
   - 缺少代码组织规范
   - 测试覆盖困难

### 影响范围
- 影响的文件数量: 7个核心文件 → 拆分为65+个模块
- 影响的功能模块: API路由、服务层、图节点、流式处理
- 是否影响API: 否（保持100%向后兼容）
- 是否需要数据库迁移: 否
- 是否向后兼容: 是（纯重构，无功能变更）

## 🔍 现状分析

### 当前代码状态

**核心问题文件分析**：

1. **app/api/main.py (4,150行)**
   - 包含：所有路由定义、依赖注入、中间件、应用配置
   - 问题：职责过多，难以导航和修改
   - 目标：拆分为路由模块 + 依赖 + 中间件

2. **app/services/auth_service.py (930行)**
   - 包含：用户认证、密码管理、会话管理、JWT处理
   - 问题：单一文件承担所有认证职责
   - 目标：拆分为多个专注服务

3. **app/services/workflow_manager.py (532行)**
   - 包含：工作流编排、状态管理、错误处理
   - 问题：逻辑复杂，难以测试
   - 目标：拆分为工作流组件

4. **app/services/hybrid_retriever.py (512行)**
   - 包含：向量检索、关键词检索、混合策略
   - 问题：多种检索策略耦合
   - 目标：拆分为独立检索器

5. **app/services/loaders.py (508行)**
   - 包含：PDF、TXT、CSV、JSON等多种加载器
   - 问题：所有加载器混在一起
   - 目标：每种加载器独立模块

6. **app/graph/streaming.py (503行)**
   - 包含：流式处理逻辑 + SSE编码
   - 问题：流式处理和编码耦合
   - 目标：分离关注点

### 问题根因

1. **快速迭代**：为了快速添加功能，代码堆积在单一文件
2. **缺少重构**：没有定期进行代码整理
3. **缺少规范**：没有明确的代码组织标准
4. **时间压力**：功能优先，质量后置

## 💡 解决方案

### 整体重构策略

采用**三阶段渐进式重构**：

#### Phase 0 (P0): 核心API和依赖拆分
- 拆分`app/api/main.py`为路由模块
- 拆分`app/api/dependencies.py`为专注模块
- **目标**: API代码减少96%+

#### Phase 1 (P1): 服务层重构
- 拆分`auth_service.py`
- 拆分`workflow_manager.py`
- 拆分`hybrid_retriever.py`
- **目标**: 服务层模块化，减少80%+

#### Phase 2 (P2): 工具和流式处理
- 拆分`loaders.py`
- 拆分`streaming.py`
- **目标**: 工具层模块化，减少85%+

### 技术选型

**重构工具**：
- AST分析工具 - 理解代码结构
- Import追踪工具 - 验证依赖关系
- 测试覆盖工具 - 确保无破坏性变更

**质量保证**：
- 单元测试 - 所有模块
- 集成测试 - 端到端验证
- 性能测试 - 确保无降低
